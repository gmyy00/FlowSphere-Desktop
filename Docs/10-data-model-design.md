## 1. 设计理念

### 1.1 统一模型

**核心原则**：所有日程本质上都是待办事项，日程是定时重复的待办。

因此本项目只保留一个核心数据模型 `Todo`，不再单独定义 Schedule。

### 1.2 模型简化

原有设计包含两个模型：
- `Todo`：待办事项
- `Schedule`：日程

新设计将两者合并：
- `Todo`：统一的待办/日程模型

**简化理由**：
1. 日程本质上是定时重复的待办
2. 减少模型复杂度，降低维护成本
3. 统一数据结构，简化存储和读取逻辑

## 2. Todo 数据模型

### 2.1 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | string | 是 | 唯一标识符，使用 UUID v4 生成 |
| `title` | string | 是 | 待办标题 |
| `description` | string | 否 | 待办的文字描述 |
| `notification` | string (ISO 8601) | 否 | 提醒时间，精确到分钟，格式：`YYYY-MM-DDTHH:MM` |
| `repeat` | enum | 否 | 重复规则（见 2.2） |
| `done` | boolean | 是 | 完成状态，默认 `false` |
| `created_at` | string (ISO 8601) | 是 | 创建时间，格式：`YYYY-MM-DDTHH:MM` |

### 2.2 字段说明

#### id
- 唯一标识符
- 使用 UUID v4 自动生成
- 格式：`xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx`

#### title
- 待办/日程标题
- 必填字段
- 不能为空字符串

#### description
- 待办的文字描述
- 可选字段
- 用于记录详细信息

#### notification
- 提醒时间
- ISO 8601 格式，精确到分钟
- 格式：`YYYY-MM-DDTHH:MM`
- 储存完整的具体时间，用于设置提醒
- 业务层根据 `notification` 和 `repeat` 组合生成展示文本

#### repeat
- 重复规则
- 枚举类型（见 2.3）
- 默认值：`none`（不重复）

#### done
- 完成状态
- 布尔类型
- 默认值：`false`
- 标记为 `true` 后停止提醒

#### created_at
- 创建时间
- ISO 8601 格式
- 格式：`YYYY-MM-DDTHH:MM`

### 2.3 Repeat 重复规则枚举

| 值 | 说明 | 示例 |
|----|------|------|
| `none` | 不重复（默认） | 单次待办 |
| `daily` | 每日重复 | 每天提醒 |
| `weekdays` | 工作日重复（周一至周五） | 工作日站会 |
| `weekly` | 每周重复 | 周会 |
| `monthly` | 每月重复 | 每月财务对账 |
| `yearly` | 每年重复 | 年度体检 |

## 3. 展示文本生成规则（业务逻辑）

展示文本由业务层根据 `notification` 和 `repeat` 字段组合生成：

### 3.1 生成规则

| repeat | 提取内容 | 组合方式 |
|--------|----------|----------|
| `none` | 完整日期时间 | 直接显示 |
| `daily` | 仅时间部分 | 每天 + 时间 |
| `weekdays` | 仅时间部分 | 每个工作日 + 时间 |
| `weekly` | 星期 + 时间 | 每周 + 星期 + 时间 |
| `monthly` | 日期 + 时间 | 每月 + 日期 + 时间 |
| `yearly` | 月 + 日 + 时间 | 每年 + 月 + 日 + 时间 |

### 3.2 示例

**输入**：
- `notification`: `2026-06-11T22:16`（星期四）
- `repeat`: `weekly`

**输出**：`每周星期四22:16`

**完整示例**：

| notification | repeat | 展示文本 |
|--------------|--------|----------|
| `2026-06-11T22:16` | `none` | 2026年6月11日星期四22:16 |
| `2026-06-11T22:16` | `daily` | 每天22:16 |
| `2026-06-11T22:16` | `weekdays` | 每个工作日22:16 |
| `2026-06-11T22:16` | `weekly` | 每周星期四22:16 |
| `2026-06-11T22:16` | `monthly` | 每月11日22:16 |
| `2026-06-11T22:16` | `yearly` | 每年6月11日22:16 |

## 5. 行为规则

### 5.1 完成状态

- `done = false` 时，按 `repeat` 规则触发系统通知提醒
- 标记为 `done = true` 后停止提醒

### 5.2 重复规则

- `repeat` 定义任务的重复频率
- `notification` 定义具体的提醒时间
- 应用层负责将两个字段组合生成展示文本
- 重复任务按周期生成下一个实例

### 5.3 日程行为

- 无"完成"概念，时间过后保留但视觉淡化显示
- 重复规则按周期生成下一个实例
- 不进行冲突检测

## 7. 与旧设计的对比

### 7.1 旧设计（两个模型）

| 模型 | 说明 |
|------|------|
| `Todo` | 待办事项 |
| `Schedule` | 日程 |

### 7.2 新设计（统一模型）

| 模型 | 说明 |
|------|------|
| `Todo` | 统一的待办/日程模型 |

### 7.3 变化总结

| 项目 | 旧设计 | 新设计 |
|------|--------|--------|
| 模型数量 | 2 | 1 |
| 字段数量 | 较多 | 精简 |
| 存储文件 | `todo.json` + `schedule.json` | `todos.json` |
| 业务逻辑 | 分散在两个模型 | 统一在一个模型 |
