# FlowSphere 项目总览

## 1. 项目定位

FlowSphere 是一个面向个人效率与团队协作的工作流平台。项目最终形态包含桌面智能助手、Web 协作平台、云端服务和 AI 能力。

当前阶段不追求一次性完成完整平台，而是采用分阶段推进：

本项目由两个旧项目方向合并而来：旧 FloatPlan 提供本地桌面待办日程、提醒、托盘和 AI 输入方向；旧 FlowSphere 提供 Web 协作平台、认证、项目任务管理和云原生工程方向。合并后的执行原则是先完成本地桌面闭环，再恢复云端协作平台，具体迁移边界见 `08-merge-strategy.md`。

| 阶段 | 目标 | 交付形态 |
|---|---|---|
| Phase 1 | 本地待办日程 Demo | Windows 桌面应用 |
| Phase 2 | 桌面智能助手增强 | 托盘、悬浮窗、主题、AI 输入 |
| Phase 3 | 云端协作平台 | Web 前端 + Go 微服务后端 |
| Phase 4 | 多端同步与提醒服务 | 账号、同步、提醒服务 |
| Phase 5 | AI 工作流能力 | 自然语言创建任务、日程和项目事项 |

## 2. 一期工程定位

一期工程优先完成本地 Demo，作为整个 FlowSphere 的最小可用产品。

一期名称建议：

```text
FlowSphere Desktop Demo
```

一期只解决一个问题：

> 用户可以在本地桌面应用中管理待办和日程，并获得基础提醒。

## 3. 目标用户

| 用户 | 需求 |
|---|---|
| 开发者本人 | 用最小成本做出可运行 Demo，后续逐步扩展 |
| 演示观看者 | 能直观看到待办、日程、提醒和本地持久化效果 |
| 未来用户 | 需要轻量的本地任务和日程管理工具 |

## 4. 产品原则

| 原则 | 说明 |
|---|---|
| 本地优先 | 一期不依赖网络、账号和服务器 |
| Demo 可演示 | 功能闭环优先于复杂架构 |
| 数据简单 | 一期使用 JSON 文件持久化 |
| 后续可扩展 | 目录结构和数据模型预留同步、AI、云端扩展点 |
| 控制范围 | 不在一期做登录、云同步、复杂重复规则和多端冲突处理 |

## 5. 一期核心闭环

```text
启动应用
  -> 查看待办和日程列表
  -> 新增待办或日程
  -> 设置提醒时间
  -> 保存到本地文件
  -> 到提醒时间弹出系统通知或应用内提醒
  -> 编辑、删除、完成待办
  -> 重启应用后数据仍然存在
```

## 6. 非目标

一期明确不做：

| 功能 | 原因 |
|---|---|
| 用户登录 | 本地 Demo 不需要账号系统 |
| 云同步 | 会引入后端、冲突处理和网络状态问题 |
| Web 平台 | 留到后续阶段 |
| Kubernetes | 一期不是云端服务 |
| 复杂 AI 能力 | 可在后续作为增强功能 |
| 日历视图 | UI 成本较高，一期用列表即可 |
| 多用户协作 | 属于云端协作平台能力 |

## 7. 成功标准

一期完成时应满足：

| 编号 | 标准 |
|---|---|
| S1 | 应用可以在 Windows 本地启动 |
| S2 | 用户可以创建待办和日程 |
| S3 | 用户可以编辑、删除、完成待办 |
| S4 | 用户可以编辑、删除日程 |
| S5 | 提醒时间到达后有可见提醒 |
| S6 | 数据能保存到本地 JSON 文件 |
| S7 | 关闭再打开后数据不丢失 |
| S8 | README 或文档能说明如何运行 Demo |
