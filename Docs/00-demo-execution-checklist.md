# Demo 阶段执行清单

## 1. 负责人结论

Demo 阶段由当前实现负责人完整交付，目标是先做出一个可以本地运行、可以验收、核心逻辑有测试保护的桌面待办日程应用。

Demo 阶段不追求完整平台化，也不把旧项目能力一次性全部搬入。旧 FlowSphere 的 Web 协作平台能力进入后续阶段，旧 FloatPlan 的 AI、主题、托盘增强和开机自启进入后续阶段。

## 2. Demo 阶段必须交付

| 编号 | 交付项 | 说明 | 验收方式 |
|---|---|---|---|
| D-01 | Python 桌面工程 | `desktop/` 下有可运行源码 | 能启动应用 |
| D-02 | 待办管理 | 新增、编辑、删除、完成、取消完成 | UI 手工验收 + 服务测试 |
| D-03 | 日程管理 | 新增、编辑、删除、状态判断 | UI 手工验收 + 服务测试 |
| D-04 | 本地持久化 | JSON 文件保存和读取 | pytest 验证 |
| D-05 | 提醒能力 | `remind_at` 到期后提醒且不重复 | pytest 验证 + UI 手工验收 |
| D-06 | 异常处理 | 文件不存在自动创建，JSON 损坏备份 | pytest 验证 |
| D-07 | 基础 UI | 双标签页：待办、日程 | 手工验收 |
| D-08 | README | 包含安装、测试、运行、验收步骤 | 文档检查 |

## 4. TDD 开发顺序

开发按先测试、后实现、再接 UI 的顺序推进。

| 顺序 | 先写测试 | 再实现 |
|---|---|---|
| T1 | StorageService 文件不存在、保存读取、JSON 损坏备份 | `core/storage_service.py` |
| T2 | Todo 创建、编辑、删除、完成、提醒候选过滤 | `models/todo.py` + `core/todo_service.py` |
| T3 | Schedule 创建、编辑、删除、时间校验、状态判断 | `models/schedule.py` + `core/schedule_service.py` |
| T4 | Reminder 到期判断、已提醒过滤、完成待办过滤、触发后标记 | `core/reminder_service.py` |
| T5 | 服务组合读写持久化 | 服务集成测试 |
| T6 | UI 接入服务 | `ui/` 和 `app.py` |

## 5. 验收清单

验收时按以下步骤执行：

1. 在 `desktop/` 安装依赖。
2. 运行 `pytest`，确认核心测试通过。
3. 启动应用。
4. 新增一个待办，填写标题、截止时间、提醒时间。
5. 编辑该待办标题。
6. 标记该待办完成，再取消完成。
7. 删除一个待办，确认列表刷新。
8. 新增一个日程，填写标题、开始时间、结束时间、提醒时间。
9. 编辑该日程标题。
10. 删除一个日程，确认列表刷新。
11. 新增一个 1 分钟后提醒的待办或日程。
12. 等待提醒弹出，确认同一记录不重复提醒。
13. 关闭应用后重新启动，确认历史数据仍存在。

## 6. 完成标准

Demo 阶段完成时必须满足：

| 标准 | 要求 |
|---|---|
| 功能闭环 | 待办、日程、提醒、持久化可完整演示 |
| 测试闭环 | 核心服务和存储测试通过 |
| 工程闭环 | 有清晰目录、依赖、运行说明 |
| 范围闭环 | 不引入云端、账号、AI、复杂主题等非 Demo 能力 |
