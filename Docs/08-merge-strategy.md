# 项目合并策略与迁移边界

## 1. 合并背景

`oldDocs` 中实际包含两条项目线：

| 来源 | 原定位 | 核心能力 |
|---|---|---|
| `oldDocs/docs01` | FlowSphere 云原生协作平台 | 用户认证、项目管理、任务协作、Go 微服务、Web 前端、Docker/Kubernetes、CI/CD、可观测性 |
| `oldDocs/docs02` | FloatPlan 本地 AI 悬浮日程助手 | Windows 桌面、托盘、悬浮窗、待办日程、提醒、主题、本地 Ollama AI 输入 |

合并后的 `Docs` 应以新的 FlowSphere 为唯一产品基准：先交付本地桌面待办日程 Demo，再逐步吸收桌面助手、云端协作、同步和 AI 能力。

## 2. 合并后的产品主线

新的 FlowSphere 不应简单叠加两个旧项目，而应按用户路径整合：

```text
个人本地效率工具
  -> 桌面智能助手
  -> 云端账号与同步
  -> 团队项目协作
  -> AI 工作流入口
```

Phase 1 已经有 Demo 雏形，当前重点是保持闭环稳定：待办、日程、提醒、JSON 持久化、基础 UI 和测试。

## 3. 能力迁移决策

| 能力 | 来源 | 合并后处理 |
|---|---|---|
| 待办、日程、提醒、JSON 持久化 | FloatPlan | 已进入 Phase 1，作为当前 Demo 核心 |
| 普通桌面窗口、双标签页 | 新 Demo | 保留，优先保证可运行和可验收 |
| 系统托盘、置顶悬浮窗、开机自启 | FloatPlan | 放入 Phase 2，不阻塞 Phase 1 |
| 主题和自定义 QSS | FloatPlan | 放入 Phase 2，先做预设主题，再做自定义 |
| 本地 Ollama 自然语言输入 | FloatPlan | 可在 Phase 2 做本地入口，复杂 AI 工作流放 Phase 5 |
| 用户认证、JWT、项目、成员、任务 | docs01 FlowSphere | 放入 Phase 3，作为 Web 协作平台主线 |
| Docker、CI/CD、Kubernetes、监控 | docs01 FlowSphere | Phase 3 先 Docker/CI，Kubernetes 和观测性后置 |
| 云端个人待办、日程、同步提醒 | 两者交叉 | Phase 4 新增 Reminder/Sync 能力，连接 Desktop 和 Web |

## 4. 当前阶段建议

Phase 1 不建议再引入 AI、托盘、主题、账号或后端。原因是这些能力都会扩大调试面，而当前 Demo 最需要的是“能稳定演示”。

建议优先补齐：

1. 明确 `desktop/README.md` 中的演示步骤和数据文件位置。
2. 保持服务层测试覆盖存储、待办、日程、提醒。
3. 清理或忽略运行产生的 `desktop/data/`、`__pycache__/` 等本地文件。
4. 在 Phase 1 验收后，再决定 Phase 2 是优先“产品化桌面体验”，还是优先“本地 AI 输入”。

## 5. 文档维护规则

`oldDocs` 只作为历史参考，不再作为开发基准。新需求、架构和计划都应写入 `Docs`。

后续改动应遵循：

| 规则 | 说明 |
|---|---|
| 一阶段一基准 | 当前开发只看对应 Phase 文档，避免旧范围回流 |
| 先改文档再扩功能 | 新增 AI、同步、后端前先更新需求和架构 |
| 保留迁移说明 | 从旧文档迁入的能力，需要说明来源和裁剪原因 |
| Demo 不承载平台目标 | Kubernetes、微服务、协作平台不应进入 Phase 1 |

## 6. 风险提示

| 风险 | 建议 |
|---|---|
| 两个旧项目目标不同 | 用阶段拆分解决，不在一个版本内同时追求桌面体验和云原生工程展示 |
| AI 能力诱惑大但依赖复杂 | 先保留 `ai_service` 扩展点，等手动创建流程稳定后接入 |
| Web 协作平台会改变数据模型 | Phase 4 前不要过早把本地 JSON 改成后端优先 |
| 文档重复导致执行混乱 | `Docs` 是唯一基准，`oldDocs` 只引用不直接执行 |
